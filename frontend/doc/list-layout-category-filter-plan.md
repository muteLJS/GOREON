## 목적

`Header.jsx`의 카테고리/브랜드 메뉴에서 넘어오는 `type` 값에 따라 `ListLayout`의 사이드 필터 목록을 다르게 보여주기 위한 사전 정리 문서다.
다음 구현 때는 이 문서를 기준으로 `type -> filterGroups` 매핑을 만들면 된다.

현재 `ListLayout`은 모든 목록에 아래 필터를 고정으로 보여준다.

```js
const manufacturerList = ["APPLE", "SAMSUNG", "ASUS", "LENOVO"];
const weightList = ["1Kg 미만", "1Kg ~ 2Kg", "2Kg ~ 3Kg", "3Kg 이상"];
const screenSizeList = ["13인치", "14인치", "15인치", "16인치", "17인치 이상"];
const gpulist = ["내장그래픽", "외장그래픽"];
```

이 조합은 노트북에는 맞지만 스마트폰, 이어폰, 프린터, PC 부품 등에는 맞지 않는다.

## 주의할 점

1. `Header.jsx`는 PC 부품 타입을 `pc-parts`로 보낸다.
   - 현재 `ListLayout.jsx`의 `TYPE_LABEL_MAP`에는 `"pc-part": "PC 부품"`만 있고 `pc-parts`가 없다.
   - 다음 구현 시 `pc-parts`도 추가하거나 타입을 통일해야 한다.
2. `Header.jsx`에는 `webcam` 타입이 없지만, `ListLayout.jsx`의 `TYPE_LABEL_MAP`에는 `webcam`이 있고 실제 데이터에도 `웹캠` 태그가 존재한다.
3. 실제 데이터 태그에는 `스피커`가 거의/전혀 없고 `웹캠`, `유선이어폰`, `무선이어폰` 등이 존재한다. Header 메뉴와 데이터 태그 간 불일치가 있다.
4. 필터 옵션은 처음에는 UI 표시용으로 넣고, 실제 필터 로직은 상품 데이터 구조와 매칭 가능한 항목부터 단계적으로 연결하는 것이 안전하다.

## 권장 filterGroups 자료구조

다음 구현에서는 아래처럼 `type`별 필터 그룹을 만들면 관리가 쉽다.

```js
const CATEGORY_FILTER_GROUPS = {
  notebook: [
    { title: "제조사", items: ["APPLE", "SAMSUNG", "LG", "HP", "LENOVO", "DELL", "ASUS", "MSI", "ACER"] },
    { title: "무게", items: ["1kg 미만", "1kg ~ 1.5kg", "1.5kg ~ 2kg", "2kg 이상"] },
    { title: "화면크기", items: ["13인치", "14인치", "15인치", "16인치", "17인치 이상"] },
    { title: "GPU 종류", items: ["내장그래픽", "외장그래픽"] },
  ],
};
```

공통 가격 필터는 모든 카테고리에 유지한다.

## AI 자연어 추천 1차 버전 연계 메모

`frontend/src/pages/Main`에 추가할 OpenAI + MongoDB 기반 자연어 추천 기능은 **1차 버전에서 상품 태그 보강 없이 구현한다.**
즉, `products_list.json`이나 MongoDB 상품 문서에 `대학생`, `휴대용`, `부모님용`, `영상편집`, `FPS` 같은 추천 전용 태그를 새로 추가하지 않는다.

### 환경변수 준비 상태

현재 `backend/.env`에는 OpenAI 연동에 필요한 아래 환경변수가 이미 준비되어 있는 것으로 확인한다.
문서에는 보안상 실제 키 값을 적지 않는다.

- `OPENAI_API_KEY`: 설정됨. 실제 값은 커밋/문서/프론트 env에 노출하지 않는다.
- `OPENAI_RECOMMENDATION_MODEL`: 설정됨. 구현 시 하드코딩하지 말고 이 값을 읽어서 사용한다.

구현자는 다음 개발 턴에서 별도 API 키 발급 작업 없이 backend env 값을 사용해 OpenAI 실제 호출 테스트를 진행할 수 있다.
단, API 키가 없거나 잘못된 환경에서도 fallback 추천이 동작하도록 방어 로직은 유지한다.

### 1차 버전 추천 기준

태그 보강 없이 아래 기존 데이터만 사용한다.

- `name`: 상품명 문자열
- `price`: 가격 문자열을 숫자로 파싱한 값
- `rating`: 평점
- `tag`: 현재 존재하는 기본 태그. 예: 브랜드, 제품군
- `priceOptions.optionName`: SSD 용량 등 옵션명

### 추천 품질 정책

1. OpenAI는 사용자의 자연어를 구조화하는 데 사용한다.
   - 예: “대학생 50만원 이하 가벼운 노트북” → `노트북`, `예산 500000 이하`, `대학생/휴대성 선호`
2. MongoDB 후보 조회는 기존 상품 데이터 기준으로만 한다.
   - 상품명, 기본 태그, 가격, 옵션명에서 매칭 가능한 조건만 사용한다.
3. DB에 없는 스펙은 추천 이유에서 단정하지 않는다.
   - 나쁜 예: “1.2kg이라 가볍습니다.” 단, DB에 무게 정보가 없으면 금지.
   - 좋은 예: “노트북 카테고리이고 예산 조건에 가까워 후보로 추천합니다.”
4. `대학생`, `휴대성`, `부모님용` 같은 용도 조건은 1차 버전에서 보조 점수로만 사용한다.
   - 명확한 데이터가 없으면 가격, 제품군, 브랜드, 평점 중심으로 추천한다.
5. 추천 결과가 애매하게 나오는 것은 1차 MVP의 허용 리스크로 둔다.
   - 기능 연결과 API 안정성을 먼저 검증한 뒤, 필요할 때 태그/스펙 데이터 보강을 2차 작업으로 진행한다.

### 백엔드 fallback 점수 예시

OpenAI API 키가 없거나 OpenAI 호출이 실패해도 1차 버전은 MongoDB 후보 기준으로 추천 카드가 나오게 한다.

```text
score =
  + 제품군/카테고리 일치
  + 브랜드 또는 상품명 키워드 일치
  + 예산 조건 일치
  + 평점 가산점
  - 예산 초과 패널티
```

### 1차 버전에서 하지 않는 것

- 상품 데이터에 추천 전용 태그 일괄 추가하지 않기
- 상세 스펙 파싱을 과도하게 만들지 않기
- 벡터 검색/임베딩 도입하지 않기
- OpenAI가 DB에 없는 스펙을 추측해서 확정적으로 말하지 않기

### 2차 개선 후보

추천 결과가 너무 넓거나 이유가 약하면 그때 아래를 검토한다.

- 자주 쓰는 용도 태그 추가: `대학생`, `게이밍`, `사무용`, `영상편집`, `부모님용`, `휴대용`, `가성비`
- 주요 스펙 태그 추가: `16GB RAM`, `SSD 512GB`, `144Hz`, `QHD`, `무선`, `블루투스`
- 상품명/옵션명에서 스펙을 추출해 별도 필드로 저장
- 벡터 검색 또는 임베딩 기반 의미 검색 도입

### 구현 시 연결 위치

- 메인 AI 입력/결과 UI: `frontend/src/pages/Main/Main.jsx`
- 메인 AI 스타일: `frontend/src/pages/Main/Main.scss`
- 추천 API 호출 유틸 후보: `frontend/src/utils/recommendations.js`
- 백엔드 추천 API 후보: `POST /api/recommendations/ai`
- MongoDB 상품 후보 조회: 기존 `/api/products` 검색 로직과 같은 기준을 재사용하되, 추천 전용 태그 보강은 하지 않는다.
